Lesson 26: Flask REST API Versioning

1. Why API Versioning?

API versioning allows evolving your API without breaking existing clients. Clients can use specific versions while new features are added in newer versions.

2. URL-Based Versioning

@app.route("/api/v1/todos", methods=["GET"])
def get_todos_v1():
  return jsonify({"version": "v1", "todos": ["task1", "task2"]})

@app.route("/api/v2/todos", methods=["GET"])
def get_todos_v2():
  return jsonify({"version": "v2", "todos": [{"id":1,"task":"task1"},{"id":2,"task":"task2"}]})
Clients can access /api/v1/todos or /api/v2/todos based on compatibility.

3. Header-Based Versioning

@app.route("/api/todos", methods=["GET"])
def get_todos_header():
  version = request.headers.get("X-API-Version", "v1")
  if version == "v1":
    return jsonify({"version": "v1", "todos": ["task1", "task2"]})
  else:
    return jsonify({"version": "v2", "todos": [{"id":1,"task":"task1"},{"id":2,"task":"task2"}]})
Clients specify version in request headers: X-API-Version: v2

4. Notes & Best Practices

  • Always maintain backward compatibility for existing clients
  • Use semantic versioning for clarity (v1, v2, etc.)
  • Document versions clearly in API documentation
  • Consider deprecating old versions after a transition period
