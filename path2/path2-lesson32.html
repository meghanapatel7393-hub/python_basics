Lesson 32: Flask REST API Scaling & Load Balancing

1. Why Scaling & Load Balancing?

Scaling ensures your API can handle increasing traffic without downtime. Load balancing distributes traffic across multiple servers for reliability and performance.

2. Vertical Scaling

Increase resources (CPU, RAM) of a single server. Easy but limited by hardware capacity.

3. Horizontal Scaling

Add multiple servers to handle requests. Requires a load balancer to distribute traffic.

4. Using Gunicorn with Multiple Workers

# Example: 4 workers
gunicorn -w 4 -b 0.0.0.0:8000 app:app
Each worker handles requests concurrently, improving throughput on a single server.

5. Load Balancing with Nginx

upstream flask_app {
  server 127.0.0.1:8000;
  server 127.0.0.1:8001;
}

server {
  listen 80;
  server_name example.com;
  location / {
    proxy_pass http://flask_app;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
  }
}
Nginx distributes requests across multiple Gunicorn processes or servers.

6. Notes & Best Practices

  • Use horizontal scaling for high traffic applications
  • Monitor server health and scale dynamically if needed
  • Combine caching, database optimization, and load balancing
  • Use containerization (Docker) for easier scaling
